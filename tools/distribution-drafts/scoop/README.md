# Scoop bucket — wheels-dev/scoop-wheels

Two manifests, mirroring the brew tap structure:

- `bucket/wheels.json` — stable, tracks `wheels-dev/wheels` GA tags
- `bucket/wheels-be.json` — bleeding-edge, tracks `wheels-dev/wheels-snapshots` pre-releases

Each uses Scoop's `checkver` + `autoupdate` to refresh URL/hash on every GA or
snapshot publish — no auto-bump workflow needed (Scoop's bucket-update bot
runs hourly across community buckets).

## Setup (you do this once)

```powershell
# Add the bucket
scoop bucket add wheels https://github.com/wheels-dev/scoop-wheels

# Install one channel
scoop install wheels        # stable
scoop install wheels-be     # bleeding-edge
```

The `bin` shim creates `wheels` on the PATH from either package — so a user can
only have one channel installed at a time. Switching:

```powershell
scoop uninstall wheels && scoop install wheels-be
scoop uninstall wheels-be && scoop install wheels
```

## How it differs from the brew tap

| Concern | Homebrew | Scoop |
|---|---|---|
| Auto-bump source | `repository_dispatch` from publish-snapshot.yml → tap workflow opens PR | `checkver` polls GitHub releases hourly, `scoop update wheels-be` self-fetches new versions |
| Hash verification | sha256 baked into the formula | sha256 baked into the manifest, regenerated by `autoupdate` block |
| First-install delay | ~5min from GA push | ~hour from GA push (next checkver cycle) |
| Maintenance overhead | 1 PR per channel per release | None |

Scoop's hourly checkver cycle is acceptable here — bleeding-edge users care
about *eventual* freshness more than minutes-of-latency, and stable users only
update on GA cuts which are infrequent enough that the latency is invisible.

## Setup checklist for the maintainer

1. Create `wheels-dev/scoop-wheels` repo (empty, public).
2. Drop `wheels.json` and `wheels-be.json` from this directory into the
   repo root under `bucket/` (the canonical Scoop layout).
3. Replace the `0000…` sha256 placeholders in `wheels.json` with real values
   computed from the v4.0.0 GA release artifacts. (Scoop's `autoupdate` block
   keeps them current after the first release; the initial values must be
   correct.)
4. Test locally:
   ```powershell
   scoop bucket add wheels-test C:\path\to\local\scoop-wheels
   scoop install wheels-test/wheels
   wheels --version    # should report 4.0.0 (stable)
   ```
5. Verify the autoupdate block by running `scoop update wheels-test/wheels`
   after a known-good GA bump. If it doesn't pick up the new version,
   the `checkver` regex needs adjusting.

## Skipping Chocolatey

We deliberately don't ship a Chocolatey package — see
[../README.md](../README.md) and the project's release-process doc for the
rationale. Existing `wheels` 0.3.x submissions on chocolatey.org are left
to bake; new versions are not submitted.
