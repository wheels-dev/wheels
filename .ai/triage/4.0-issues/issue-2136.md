# Issue #2136: Database Incompatibility in test suite

## Verdict
FIX NOW — distinct from #2106/#2110. Do NOT merge (different file, different surface). Consolidate into a single "DB compat" epic only for tracking.

## Summary
The **app-level** `tests/populate.cfm` (76 lines, used by the Wheels Web/app test harness via `tests/TestRunner.cfc`) only branches DB-specific DDL for MySQL/MariaDB/PostgreSQL. On SQLite, Oracle, CockroachDB (and H2), it emits SQL Server defaults (`int NOT NULL IDENTITY`, `datetime`, `text`) and blows up at the populate step. The **core** `vendor/wheels/tests/populate.cfm` already handles all these engines correctly — this issue is narrowly about the app-level file, which the author explicitly cites.

## Root cause
Not a dup of #2106 or #2110.

- **#2106 — CockroachDB core**: Framework code issue in `vendor/wheels/` (e.g. `ON CONFLICT ... RETURNING` bulk ops, advisory locks used in migrator). Surfaces in `/wheels/core/tests`. Owned by model/migrator layer.
- **#2110 — Core tests across all DBs**: Symptom-level report against `/wheels/core/tests`. Likely a mix: some are the #2106 CockroachDB subset, some may be regressions post-4.0 refactors; needs triage of its own.
- **#2136 — app `tests/populate.cfm`**: Pure test-fixture DDL gap. Wrong file set, wrong test runner (`/wheels/app/tests`, not `/wheels/core/tests`). The author's screenshot and "Review errors originating from /tests/populate.cfm" pin the scope unambiguously.

Why the app file is deficient (compared to core):
1. Default `local.identityColumnType = "int NOT NULL IDENTITY"` targets SQL Server; on SQLite/Oracle/CockroachDB it's invalid DDL.
2. Only three DB branches: MySQL/MariaDB (AUTO_INCREMENT + InnoDB) and PostgreSQL (SERIAL). Missing: SQLite (`INTEGER` PK autoincrement), Oracle (`INTEGER GENERATED BY DEFAULT AS IDENTITY`), CockroachDB (`SERIAL`), H2, Microsoft SQL Server (explicit branch, matching current default).
3. Hardcoded `datetime` column type — invalid on PostgreSQL (`timestamp`), Oracle (`TIMESTAMP`), SQLite (`TEXT`).
4. Hardcoded `text` column type — invalid on Oracle (`VARCHAR2(n)` / `CLOB`).
5. `int NULL` + `int DEFAULT 0 NOT NULL` for FK/counter columns — OK on most, but Oracle prefers `NUMBER(10)`.
6. The DB-detection block already exists (lines 2-15) but is truncated compared to the core populate; the fix is to port the core's pattern down.

The app populate is short enough (2 tables: `c_o_r_e_authors`, `c_o_r_e_posts`) that the fix is mechanical — adopt the same `local.identityColumnType / dateTimeColumnType / intColumnType / textColumnType / storageEngine` dispatch table used in `vendor/wheels/tests/populate.cfm` lines 20-79.

## Files to change

1. **`tests/populate.cfm`** — the file the issue names.
   - Expand the DB-detection block (currently lines 2-15) to cover: `microsoftsqlserver`, `mysql`/`mariadb`, `h2`, `postgresql`, `cockroachdb`, `oracle`, `sqlite`. Mirror the variable set from `vendor/wheels/tests/populate.cfm` (`identityColumnType`, `dateTimeColumnType`, `intColumnType`, `textColumnType`, `floatColumnType`, `binaryColumnType`, `storageEngine`).
   - Replace hardcoded `int`, `datetime`, `text` in the two `CREATE TABLE` statements with the corresponding `#local.*ColumnType#` tokens.
   - Keep the existing `DROP TABLE IF EXISTS` guard — works on MySQL/Postgres/MariaDB; for SQL Server and Oracle wrap in `cftry`/`cfcatch` (already wrapped) which is sufficient since neither supports `IF EXISTS` universally. Confirm with a quick run.
   - `application.wo.model("author").create(...)` calls need no changes — ORM handles types.

2. **`.github/workflows/compat-matrix.yml`** — no change required; already fans out across `mysql,postgres,sqlserver,cockroachdb,oracle,sqlite` (+ `h2` on Lucee). This workflow is our verification surface.

3. **`CLAUDE.md`** — no change required. Cross-engine docs already live in `.ai/wheels/cross-engine-compatibility.md`; the app-level populate isn't called out today and adding a note is low value.

4. (Optional, if adopted elsewhere) **no new helper component** — the DB-detection block is short and duplication between app + core is clearer than a third shared CFC that both include. If we ever grow a third populate, extract to `tests/_shared/dbTypes.cfm` and `cfinclude` from both.
