## The activation model

On application startup, Wheels runs `vendor/wheels/PackageLoader.cfc`. It scans `vendor/` for every subdirectory that contains a `package.json`, skips `vendor/wheels/` (the framework core itself), resolves the dependency graph across whatever it finds, and loads each package in topological order. There is no separate registration step â€” presence of a directory under `vendor/` with a valid manifest *is* the activation.

This inverts the classic plugin pattern. Instead of a registry that names every installed plugin, the filesystem layout is the registry. To add a package, you put its files where the loader looks. To remove one, you take them away.

`packages/` in the repo is a **staging area**. Files there are never loaded. Activation means getting a copy (or symlink) of a package into `vendor/` under its own directory.

<Aside type="caution">
  Wheels 4.0 does not yet ship a `wheels package install` command. Activation is manual â€” `cp -r` or a symlink. This is tracked as a framework gap; once the CLI surface ships, this page will document the install command as the canonical workflow.
</Aside>

Manual activation:

```bash title="your shell"
cp -r packages/hotwire vendor/hotwire
wheels reload
```

Symlink for active development (edits in `packages/` take effect immediately):

```bash title="your shell"
ln -s ../../packages/hotwire vendor/hotwire
wheels reload
```

Deactivation is equally manual â€” delete the `vendor/<name>` directory and reload:

```bash title="your shell"
rm -rf vendor/hotwire
wheels reload
```

The reload step re-runs `PackageLoader`, which rediscovers what's in `vendor/` and reconciles the mixin, service, and middleware tables with the new state.

## First-party packages

Four packages ship in `packages/` as of Wheels 4.0. All are optional; the framework core runs fine with none of them activated.

- **`hotwire/`** (`wheels-hotwire`) â€” Turbo Drive, Turbo Frames, Turbo Streams, and Stimulus integration for server-rendered UI. Mixes into controllers and views. Used heavily in the tutorial app.
- **`basecoat/`** (`wheels-basecoat`) â€” UI component helpers styled with Tailwind CSS. shadcn/ui-quality forms, buttons, and cards without React. Mixes into controllers and views.
- **`sentry/`** (`wheels-sentry`) â€” Sentry.io error tracking with framework-aware context enrichment. Captures exceptions with request, user, and route context. Mixes into controllers.
- **`legacyadapter/`** (`wheels-legacy-adapter`) â€” Backward-compatibility shim for Wheels 3.x plugins. Deprecation logging, API adapters, and a scanner that flags 3.x patterns you should modernize. Mixes into controllers.

A fifth candidate, `wheels-i18n`, lives as a plugin at [`wheels-dev/wheels-i18n`](https://github.com/wheels-dev/wheels-i18n) on GitHub and is planned for conversion to a first-party package in a future release.

## The `package.json` manifest

Every package declares a `package.json` at its root. The loader parses it before instantiating any CFC, so a manifest error isolates cleanly â€” the package is skipped, but the rest of the app boots.

```json title="packages/sentry/package.json"
{
    "name": "wheels-sentry",
    "version": "1.0.0",
    "author": "PAI Industries",
    "description": "Sentry error tracking for Wheels with framework-aware context enrichment",
    "wheelsVersion": ">=3.0",
    "provides": {
        "mixins": "controller",
        "services": [],
        "middleware": []
    },
    "dependencies": {}
}
```

Field reference:

- **`name`** â€” the package name as you'd reference it in docs or a future registry. Usually matches the directory but doesn't have to (`wheels-sentry` lives in `sentry/`). Required.
- **`version`** â€” semver. Required. Used in load logs and by future install tooling.
- **`author`** â€” free-form string. Not validated.
- **`description`** â€” one-line description shown in logs and tooling.
- **`wheelsVersion`** â€” compatibility range (npm semver syntax). Declared as a constraint against the framework version; mismatches are logged.
- **`provides.mixins`** â€” comma-delimited target list: `controller`, `view`, `model`, `global`, `none`. **Default is `none`** â€” a package with no `provides.mixins` field contributes no mixed-in methods. This is the explicit opt-in model and is a deliberate break from legacy plugins, which defaulted to `global` and made reasoning about method provenance hard.
- **`provides.services`** â€” reserved for services the package registers with the DI container. Currently populated by the package's own `register()` hook rather than read from the manifest; the field is here for future tooling.
- **`provides.middleware`** â€” array of middleware entries the loader appends to the global pipeline. Each entry is `{"component": "vendor.mypackage.MyMiddleware", "options": {...}}`.
- **`dependencies`** â€” map of other package names to version ranges. The loader resolves these into a load order (topological sort) so a dependent package sees its dependencies already loaded.

The four `provides.mixins` targets map to where a package's public methods become callable:

- `controller` â€” methods available on every controller instance (and in controller filters)
- `view` â€” methods available in every view and partial
- `model` â€” methods available on every model
- `global` â€” all three at once (use sparingly; it's the old plugin default and the one we're steering away from)
- `none` â€” no mixin; the package is service/middleware-only

## Writing your own package

A package is a directory with a `package.json` and at least one CFC. The loader's convention is that the main CFC's filename matches the directory name â€” `packages/myfeature/MyFeature.cfc` â€” and gets instantiated via `CreateObject("component", "vendor.myfeature.MyFeature").init()`. If that file doesn't exist, the loader falls back to the first `.cfc` it finds in the directory.

A minimal layout:

```
packages/myfeature/
  package.json                     # manifest (required)
  MyFeature.cfc                    # main entry â€” matches directory name
  middleware/MyMiddleware.cfc      # optional â€” referenced in provides.middleware
  tests/                           # optional â€” runs via the core test runner
```

A minimal manifest:

```json title="packages/myfeature/package.json"
{
    "name": "my-feature",
    "version": "0.1.0",
    "description": "Adds a thing to Wheels",
    "wheelsVersion": ">=4.0",
    "provides": {
        "mixins": "controller"
    }
}
```

A minimal entry CFC:

```cfm {test:compile} title="packages/myfeature/MyFeature.cfc"
component output="false" {

    public any function init() {
        return this;
    }

    /**
     * Public methods become mixins on the targets declared in provides.mixins.
     * This one lands on every controller because the manifest says "controller".
     */
    public string function myHelper(required string name) {
        return "Hello, " & arguments.name;
    }
}
```

After copying `packages/myfeature/` to `vendor/myfeature/` and reloading, every controller has `myHelper()` available. The framework collects public methods from the package instance and merges them into the application mixin tables â€” the same machinery that runs for plugins, but scoped to the targets you named in the manifest.

Lifecycle hook names the loader specifically skips when collecting mixins: `init`, `onPluginLoad`, `onPluginActivate`, `register`, `boot`. If you name a method any of those, it won't be mixed in â€” the loader treats them as package infrastructure. `register(container)` and `boot(app)` are recognized as DI service-provider hooks if the CFC declares `implements="wheels.ServiceProviderInterface"`.

## Per-method mixin overrides

The manifest-level `provides.mixins` sets the default for every public method. If one method needs a different target, annotate the method with a `mixin` metadata attribute â€” the loader reads it via `GetMetadata()` and overrides the package default for that method only.

```cfm {test:compile} title="packages/myfeature/MyFeature.cfc"
component output="false" {

    public any function init() {
        return this;
    }

    // This method follows the manifest default (controller).
    public string function controllerOnlyHelper() {
        return "I'm on controllers";
    }

    // This method overrides to view-only, even if the manifest says "controller".
    public string function viewOnlyHelper() mixin="view" {
        return "I'm on views";
    }
}
```

## Error isolation

Every package loads inside its own try/catch. A package that throws during manifest parsing, CFC instantiation, or mixin collection is recorded in the loader's `failedPackages` array and logged to the application log. The rest of the app â€” framework core, other packages, application code â€” continues loading normally.

Common failure modes and what happens:

- **Malformed `package.json`** â€” logged as a manifest error, package skipped.
- **Missing required manifest fields (`name`, `version`)** â€” same: logged and skipped.
- **Dependency not installed** â€” the dependency-resolution phase records the error before any CFC is instantiated. Dependent packages are excluded from the load order.
- **Circular dependencies** â€” surfaces as a graph error at resolution time; all packages in the cycle are excluded with a clear log message.
- **Exception during CFC `init()`** â€” caught, logged with stack, package skipped. Mixins are never partially applied.

The framework's contract is that activating a broken package cannot take down a working app. Check the application log after a deploy to see which packages loaded, which were skipped, and why.

## Testing a package

Package tests live under `packages/<name>/tests/`. Each first-party package ships its own suite. Once activated in `vendor/`, you can run a package's tests through the core test runner by pointing `directory` at the package subdirectory:

```bash title="your shell"
curl "http://localhost:60007/wheels/core/tests?db=sqlite&format=json&directory=vendor.sentry.tests"
```

This compiles and runs every spec under `vendor/sentry/tests/` and nothing else â€” the same mechanism the framework uses for its own suite, scoped to one package. Any spec that extends `wheels.WheelsTest` works here without extra wiring.

## Dependency resolution

When a package's manifest declares `dependencies`, the loader builds a dependency graph and loads in topological order â€” dependencies first, dependents later. A package with no dependencies can load anytime; a package that declares `"dependencies": {"hotwire": ">=0.1.0"}` is guaranteed to see the Hotwire mixins already installed before its own `init()` runs.

If a declared dependency isn't present in `vendor/`, or if two packages form a cycle, the loader records the error and excludes every package involved in the failure â€” again, without taking down the app.

## Publishing

Wheels 4.0 has no centralized package registry yet. The current distribution story:

- **First-party packages** â€” ship inside the `packages/` directory of the Wheels repo. You get them by having the framework.
- **Third-party packages** â€” published as their own GitHub repos. Users clone or download the repo, then copy the contents under `packages/` in their app, then activate into `vendor/`. Some packages ship an install script; others document a manual clone.

Several strong candidates for first-party promotion already exist as 3.x-era plugins. `wheels-i18n` is the first flagged for conversion â€” a ported, package-shaped version will replace the CLAUDE.md note in a future release.

## The future of package install

Framework gap #2 tracks adding a proper CLI surface:

- `wheels package install <name>` â€” fetch a package from a registry or URL, verify it, place it in `packages/`, and activate into `vendor/`.
- `wheels package list` â€” show installed and activated packages with versions.
- `wheels package activate <name>` / `wheels package deactivate <name>` â€” manage the `vendor/` symlink or copy without the user running `cp` and `rm`.

Until those commands land, scripting `cp -r packages/<name> vendor/<name>` in your app's deploy pipeline (or committing the `vendor/<name>` copy directly to your app repo) is the best practice. Both approaches are stable â€” the activation model itself isn't expected to change when the CLI arrives, only the ergonomics.
